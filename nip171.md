Tiny Groups

Features and Limitations:
 - each group has one and only one group owner (a regular nostr account)
 - each group has its own private-public key. Therefore it has a profile, custom relays and everything a regular account has.
 - the group owner is the one that generates the group private-public key and is the only one that can add or remove members
 - any group member can reveal messages passed in the group while he was a member (there is no way to stop this)
 - a member can send messages in the group ONLY as long as he is a member


Group management:
 - the group owner starts a group by generating a private-public key for the group (this is the group identity)[1]
 - when the group is initialized and each time a member is removed, the group owner MUST generate a new chat-session-private-key and it MUST share it to the latest group members (see below the event)
 - the owner can invite users to the group by sending to each user a gift-wrapped (NIP-59) event containing a  `"kind": 1014` event of this form:
 ```
 {
  "id": "<usual hash>",
  "pubkey": "<group-pubkey>",
  "created_at": "<current-time>",
  "kind": 1014,
  "tags": [
    ["p", "<group-member-pubkey>"]
  ],
  "content": "<chat-session-private-key-hex>",
  "sig": "<signed by group-privatekey>"
}
```

User send:
 - when the user receives a `"kind": 1014` event from a public key it should treat that public key as a group [2]
 - when the user has the `chat-session-private-key` it can send messages to the group
 - the user can send messages to the group using NIP-17 with the following constraints:
   - the `gift-wrap` event (`"kind": 1059`) is sent to the `chat-session-public-key` (NOT to the `group-public-key`)
   - the `seal` event (`"kind": 13`) is sent to the `chat-session-public-key` and MUST contain these tags:
   ```
   "tags": [
    ["invited_at": "<`created_at` from the kind 1014 event>"],
    ["invitation_proof": "<`sig` from the kind 1014 event>"]
   ]
   ```

User receive:
 - the user MUST use the latest `chat-session-public-key` in order to listen for gift-wrap` events (NIP-17) for that group
 - after the user unrwapps the `gift-wrap` event (`"kind": 1059`) it MUST cheeck that the invitation proof is valid

 - if the invitation proof is not valid then the event MUST be dropped, this is not a member of the group


Notes:
- [1] the owner MUST save and and keep this private key secret
- [2] each client can decide how it wants to handle groups. The most common one is that the group is displayed as a regular chat
- messages sent by `chat-session-public-key` should be ignored by clients
